比谁accept()快

不知道为什么国内技术第一的开源社区会去反驳一个很不合理的benchmark。那个benchmark竟然用了Erlang臭名昭著的io:format，莫名其妙地把socket设置成active，还不自量力地和C++去比速度，C++代码还是Boost库文档里的例子。

而国内技术第一的开源社区给出的benchmark实际上就是比accept()。所以要把accept的process的priority设置成max才行。另外，Erlang默认的port数量设得比较小，需要把ERL_MAX_PORTS改得大一点。

国内技术第一的开源社区的benchmark就一次10000个连接，结束得太快了。我又懒得改ulimit上限，同时只有4000个就只有4000个吧，总数设成80万个连接。不会C++，就用C写了个。

C版本的结果

real	1m34.921s
user	0m0.498s
sys	1m9.674s

Boost/ASIO版本的结果

real	1m34.972s
user	0m0.495s
sys	1m10.013s

Erlang版本的结果

real	1m39.437s
user	0m0.520s
sys	1m11.655s

可以看到，区区80万个连接，Erlang就比C++慢了5秒。对于每天HTTP请求数动辄上十亿的巨型网站来说，每80亿个连接，Erlang仅accept()就会比C++多花5万秒啊，一天也就8万多秒，哪里还有时间来运行业务逻辑啊。

整天摆弄micro benchmark，不去探索不去尝试的弊病，Rob Pike早就吐槽过。而且，这对推广来说一点好处都没有。
